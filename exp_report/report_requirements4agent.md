# report_requirements4agent

本文件定义后续 agent 在本仓库撰写/更新实验报告的强制规范。若用户给出新反馈，必须优先更新本文件，再按新规范更新对应 report。

## 1. 报告文件命名与位置
- 所有实验报告统一放在 `exp_report/`。
- 文件名格式：`report_<config_name>.md`。
- 每个配置至少保留一份主报告，长期追加“更新记录”。

## 2. 每份报告必须包含的章节
- `实验标识`：配置路径、实验名、报告日期、主要基线报告。
- `实验目标（希望得到的结论）`：明确本实验要验证的问题。
- `Config 快照（关键字段）`：以 yaml 片段记录关键超参。
- `日志与产物路径`：至少包含主 `log.txt` 路径。
- `与基线报告的主要改动`：说明基于哪份历史报告、改了什么。
- `当前结果`：从日志抽取 step/loss/eval 等关键指标。
- `阶段性结论`：当前可得结论与风险。
- `更新记录`：按日期追加，不能覆盖历史记录。

## 3. 日志读取与更新规则（强制）
- 每次“更新 report”前，必须重新读取对应日志文件（至少 `log.txt`）。
- 更新时必须同步刷新：
  - 最新 step
  - 最新训练损失摘要（或最近若干条统计）
  - 最新评估指标（如存在）
  - 是否出现报错/中断
- 若正式日志尚不存在，可写“未开始/待生成”，但需给出目标日志路径。

## 4. 对比规则（强制）
- 每份新报告必须写清“主要基于哪一个旧报告”。
- 必须列出“主要改动项”（建议按参数 diff 列举）。
- 若同名报告多次更新，需要与“上一次更新”做差异说明。

## 5. 代码联动规则（强制）
- 当用户要求“写代码并同步更新 report”时：
  1) 先读取相关代码改动（涉及配置/训练脚本/损失逻辑）。  
  2) 再读取最新日志。  
  3) 最后同步更新对应 report 的“主要改动”“当前结果”“更新记录”。
- 不允许只改代码不更新报告（若用户明确要求联动更新）。

## 6. 本项目当前约定（来自用户反馈）
- `qwen3_vit_base_sae_vit_decoder.yaml` 作为原始 baseline，默认不改。
- `hf_dim=0` 正式实验使用独立配置：`deco-sae/qwen3_vit_base_sae_hf0_vit_decoder.yaml`。
- `hf_dim=0` 正式日志命名使用 `...hf_dim0/log.txt`，不将 `smoke` 命名作为正式实验名。

## 7. 反馈驱动更新机制
- 用户对报告格式/字段提出新要求时，agent 必须：
  - 先修改本文件对应条款；
  - 再批量修正受影响的 report；
  - 在 report 的更新记录里注明“按新规范调整”。
